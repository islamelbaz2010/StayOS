## Problem

No API style is defined for the StayOS platform. This blocks:
- Frontend-backend contract design
- API documentation standards
- Error handling conventions
- Versioning strategy
- Mobile app integration (field staff app — FC-05)

The choice of API style affects developer experience, client complexity, and long-term maintainability.

---

## Options Evaluated

### Option A: REST with OpenAPI 3.0

| Dimension | Assessment |
|-----------|-----------|
| **Cost** | Zero. FastAPI generates OpenAPI spec automatically from type annotations. |
| **Scalability** | REST is stateless — scales horizontally without affinity. |
| **Operational Complexity** | Low. Industry standard. Every HTTP client speaks REST. |
| **Security** | JWT in Authorization header. HTTPS enforced. Standard security tooling (WAF, rate limiting) works with REST. |
| **Performance** | Minimal overhead. HTTP/1.1 or HTTP/2. No query parsing overhead. |
| **Future Global Expansion** | REST is universal. Works with any client in any region. |

**Pros**: FastAPI generates OpenAPI 3.0 documentation automatically — zero documentation effort beyond type annotations. REST is the universal API style — Next.js, Android/iOS field app, and future third-party integrations all speak HTTP REST natively. Standard tooling: curl, Postman, httpie. No client library required.

**Cons**: Over-fetching and under-fetching — a listing page may need to call multiple endpoints (listing details + availability + pricing). REST versioning requires URL or header conventions.

---

### Option B: GraphQL

| Dimension | Assessment |
|-----------|-----------|
| **Cost** | Requires graphene-python or Strawberry library. Additional schema definition layer. |
| **Scalability** | Good. Persisted queries reduce bandwidth. N+1 query problem requires DataLoader — adds complexity. |
| **Operational Complexity** | High. Schema-first development. Resolver logic. DataLoader for batching. Schema versioning is implicit (field deprecation). |
| **Security** | Query depth limiting required to prevent DOS via deeply nested queries. Introspection must be disabled in production. |
| **Performance** | Can reduce over-fetching. DataLoader prevents N+1 on relationships. But query parsing adds overhead. |
| **Future Global Expansion** | Works globally. But operationally complex for small team. |

**Pros**: Clients fetch exactly what they need — reduces bandwidth on low-speed Egyptian mobile networks. Single endpoint. Strongly typed schema.
**Cons**: Significant additional complexity. No automatic OpenAPI documentation. Query depth limiting is a security concern. N+1 query problem requires DataLoader discipline — adds to development time. Over-engineered for Phase 1's straightforward CRUD API surface.

---

### Option C: tRPC (TypeScript RPC)

| Dimension | Assessment |
|-----------|-----------|
| **Cost** | Zero if TypeScript backend. Incompatible with Python backend. |
| **Scalability** | Good for TypeScript mono-repo. |
| **Operational Complexity** | Low in TypeScript ecosystem. |
| **Security** | Same as REST. |
| **Performance** | Comparable to REST. |
| **Future Global Expansion** | TypeScript-only. |

**Pros**: End-to-end type safety between frontend and backend.
**Cons**: Requires TypeScript backend. ADR-002 chose Python/FastAPI. tRPC is incompatible with Python. Immediately rejected.

## Decision

**REST API with OpenAPI 3.0 (auto-generated by FastAPI)**

### API Design Conventions

**Base URL**: `/api/v1/` — version in URL path for explicit versioning.

**Endpoint naming**: Resource-based, plural nouns.
```
GET    /api/v1/listings              → search/list listings
GET    /api/v1/listings/{id}         → listing detail
POST   /api/v1/listings              → create listing (host)
GET    /api/v1/bookings              → guest booking history
POST   /api/v1/bookings              → create booking
GET    /api/v1/bookings/{id}         → booking detail
POST   /api/v1/bookings/{id}/confirm → confirm booking (after payment)
GET    /api/v1/tickets               → ops ticket list
PATCH  /api/v1/tickets/{id}          → update ticket status
```

**HTTP methods**: GET (read), POST (create), PUT (full replace), PATCH (partial update), DELETE (soft delete — no hard delete).

**Error format** (standardized across all endpoints):
```json
{
  "error": {
    "code": "BOOKING_CONFLICT",
    "message": "The selected dates are no longer available.",
    "message_ar": "التواريخ المحددة لم تعد متاحة.",
    "details": {}
  }
}
```

**Arabic error messages**: All error responses include both `message` (English) and `message_ar` (Arabic) fields — required for Arabic-first UX (DEC-003).

**Pagination**: Cursor-based for listing search (performance at scale); offset-based for admin/backoffice lists.

**Authentication**: JWT Bearer token in `Authorization: Bearer <token>` header. See ADR-006.

**Versioning strategy**: URL versioning (`/api/v1/`, `/api/v2/`). No header-based versioning — harder to test and document.

**Rate limiting**: API gateway layer (AWS ALB + WAF rules). OTP endpoints: 3 requests per phone per 15 minutes. Search: 60 requests per IP per minute.

### Next.js BFF Layer

Next.js API routes (`/app/api/`) serve as Backend-for-Frontend for browser clients:
- Combine multiple FastAPI calls into a single page load (listing detail + availability + host profile)
- Handle cookie-based session management for browser clients (FastAPI handles Bearer tokens for mobile)
- Never expose FastAPI backend URL to browser clients

Field staff mobile app calls FastAPI directly via Bearer token.

---

## Decision Rationale

1. **FastAPI generates OpenAPI automatically**: Zero documentation effort. Type annotations in Python become API docs. This is a significant developer productivity gain for a small team.

2. **REST is the right abstraction for Phase 1**: The StayOS API surface is mostly CRUD — listings, bookings, tickets, users. These map naturally to REST resources. GraphQL's advantages (flexible querying, reduced over-fetching) require a mature schema and DataLoader discipline that is premature at Phase 1.

3. **tRPC is incompatible**: Python backend makes tRPC immediately invalid.

4. **URL versioning is explicit**: `/api/v1/` and `/api/v2/` can coexist. Header-based versioning is invisible to logs, proxies, and caches.

5. **Arabic error messages from day one**: ENGINEERING_RULES.md §9 requires Arabic RTL support from initial commit. Error messages are the first Arabic UX surface — both `message` and `message_ar` fields enforced by Pydantic error response model.

---

## Rejected Alternatives

| Alternative | Reason Rejected |
|------------|----------------|
| GraphQL | Significant additional complexity; N+1 problem requires DataLoader; no auto-documentation; over-engineered for Phase 1 CRUD surface |
| tRPC | Requires TypeScript backend; incompatible with Python/FastAPI (ADR-002) |
| gRPC | Binary protocol; not browser-native; adds client stub generation complexity; overkill for Phase 1 |

---

## Migration Cost

**Phase 3 → partial GraphQL**: If frontend teams need flexible querying (Phase 3+ with multiple consumer apps), add a GraphQL gateway (Apollo Server or Strawberry) in front of the REST API. REST endpoints remain unchanged. Migration cost: additive only.
